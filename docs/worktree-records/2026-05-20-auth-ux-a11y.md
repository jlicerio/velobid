## Findings

### P1 — Signup labels are visual only

`/signup` fields had visible label text, but text inputs lacked stable `id` values and labels lacked `htmlFor` associations. Inputs were primarily discoverable by placeholder text rather than associated labels.

### P1 — Signup fields lacked autocomplete

Signup inputs did not include autocomplete metadata for organization/name/email/new-password/tel/location, degrading browser autofill and password manager behavior.

### P1 — Signup terms link pointed to `#`

The `terms and conditions` link in `/signup` used `href="#"` instead of navigating to the newly restored `/terms` route.

### P2 — Login labels and error feedback needed semantics

Login fields had autocomplete from PR #9 but still lacked stable `id`, `name`, and label associations. Invalid-login feedback was visible but not exposed as an alert/live message or associated with inputs.

## Fixes Applied

### `frontend/src/pages/LoginPage.tsx`

- Added `id` and `name` to User ID and Password inputs.
- Added `htmlFor` on labels.
- Preserved autocomplete:
  - `username`
  - `current-password`
- Added `role="alert"` and `id="login-error"` for invalid credentials.
- Added `aria-invalid` and `aria-describedby` linking both fields to `login-error` when an error is present.

### `frontend/src/pages/SignupPage.tsx`

- Added stable field IDs:
  - `company-name`
  - `primary-contact`
  - `admin-email`
  - `signup-password`
  - `phone`
  - `location`
  - `accept_terms`
- Added `htmlFor` on all visible labels.
- Added autocomplete values:
  - company: `organization`
  - primary contact: `name`
  - admin email: `email`
  - password: `new-password`
  - phone: `tel`
  - location: `address-level2`
- Added field-level `aria-invalid` and `aria-describedby` for validation errors.
- Changed the terms link to React Router `<Link to="/terms">`.
